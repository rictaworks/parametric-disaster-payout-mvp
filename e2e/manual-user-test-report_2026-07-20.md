### バグ#3の根本原因（特定済み）

`config/application.rb` で `config.api_only = true` が設定されているため、Rails標準のミドルウェアの一部
（Cookies・Session・Flash・**Rack::MethodOverride** 等）がデフォルトで除外される。
PR #75（`b809e58`）で Cookies・Session・Flash は手動で re-add されているが、**`Rack::MethodOverride` だけ
re-add されていない**。

管理画面の「支払完了（模擬）にする」「無効化」ボタンは `button_to ..., method: :patch` で実装されており、
ブラウザは実際には `POST` + 隠しフィールド `_method=patch` を送信する。通常はここを
`Rack::MethodOverride` が拾って実質PATCHとして扱うが、本番にこのミドルウェアが存在しないため生のPOSTのまま
処理され、`patch "payouts/:id/complete"` のルートにマッチせず404になる。

検証済みの裏付け：
- フォーム自体は正しい（CSRFトークン・`_method=patch`・action URL は正常）
- `form.submit()` を直接JSで呼んでも同じ404（クリックシミュレーションの問題ではない）
- 同じ管理画面でもPOST限定のフォーム（模擬イベント注入）は正常動作 → PATCH変換だけが壊れている（`reset`は本番で404のため未実施・本根拠には含めない。61行目・70行目を参照）
- Railway本番は最新コミット（PR #86時点）を反映済み

**推奨対応**: `config/application.rb` または該当コントローラーで `Rack::MethodOverride` を re-add するか、
`button_to` を `method: :post` の専用エンドポイントに変更する。Issue化を推奨。

### 新規発見：Issue #98（ログアウトAPIが503で常に失敗）

Issue #94（ログアウト時の表示文言修正）を確認するためログアウトボタンを操作したところ、
`DELETE /api/v1/session` が**100%再現するかたちで503**を返し、PR #96（表示文言修正）の成功パスに
到達できないことを発見した。

原因はRailsバックエンド側ではなく、Next.js側のBFFプロキシ（`src/frontend/lib/api-proxy.ts`）にある：

- Rails側（`Api::V1::SessionsController#destroy`）は正しく `204 No Content` を返している（Railwayログで確認）
- Fetch APIの仕様上、ステータス204/205/304では `Response` コンストラクタに非nullの`body`を渡すと
  `TypeError: Response constructor: Invalid response status code 204` が発生する（Vercel Edge Runtimeログで実際のスタックトレースを確認）
- `api-proxy.ts` はレスポンスボディを常に文字列（204の場合は空文字列 `""`）として `new Response(proxiedBody, { status: backendResponse.status, ... })` に渡しており、空文字列は「非null」なためこの制約に抵触しクラッシュしていた
- クラッシュがVercel Edge Function側で発生するため、ブラウザ・利用者からは503として観測される

**影響**：本番のログアウト機能は、PR #96の対象であるメッセージ表示バグ（Issue #94）とは独立に、
より根本的なレイヤーで常に失敗する状態にある。したがって **PR #96は現時点で本番検証不可能**（成功パス自体に到達できないため）。

Issue #98として起票済み：https://github.com/rictaworks/parametric-disaster-payout-mvp/issues/98
（修正案として、ステータス204/205/304の場合は `body` に `null` を渡すよう分岐する変更を提案済み）

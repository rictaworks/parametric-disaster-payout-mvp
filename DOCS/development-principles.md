# 開発原則

## 基本原則
- **YAGNI**: いま必要ない機能は作らない
- **KISS**: できるだけ単純に作る
- **DRY**: 重複を避ける。ただし無理な共通化はしない
- **SOLID**: 変更しやすく責務の明確な設計を優先する

## 設計指針
- 関心の分離を徹底する
- 高凝集・低結合を目指す
- 継承よりコンポジションを優先する
- 実装よりインターフェースに依存する
- surprise の少ない命名・構造にする

## 実装方針
- 小さく作って早く確認する
- 失敗は早く検出する（Fail Fast）
- コメントより読みやすいコードを優先する
- 触った場所は前より少し良くして返す（Boy Scout Rule）

## 判断基準
迷ったら次の順で優先する:
1. シンプルか
2. いま本当に必要か
3. 責務が分かれているか
4. 将来の変更に耐えやすいか
5. 重複を減らせているか

## セキュリティ原則
- **シークレットはファイルではなくプラットフォームの環境変数（Railway Variables・Vercel Environment Variables 等）で管理する**
- フレームワーク固有のシークレットファイル（Rails の `config/master.key` 等）は初回コミット前に `.gitignore` に含まれていることを必ず確認する
- `.gitignore` の設定より先にシークレットファイルをコミットしない（順序厳守）
- 初回コミット前に `git status` でシークレットファイルがステージングされていないことを確認する

## 避けること
- 使う予定ベースの先回り実装
- 巨大な関数 / 巨大なクラス
- 意味の薄い抽象化
- コメントがないと読めないコード
- 1つの変更で多箇所修正が必要な設計
